Accessibility
Accessibility requirements and guidelines for building inclusive Sprout experiences — covering semantic markup, keyboard navigation, color contrast, ARIA, and touch targets.

Core Requirements

Every Sprout screen and component must meet these five accessibility standards.

1

Semantic HTML

  • Use <button> for actions, <a> for navigation — never <div onclick>
  • Wrap page regions with <nav>, <main>, <header>, <section>, and <footer>
  • Use <table> with <thead> and <th> for tabular data
  • Use <ul> / <ol> for lists — assistive tech announces item counts
  • Form inputs must have associated <label> elements or aria-label
2

Keyboard Navigation

  • All interactive elements must be reachable via Tab key
  • Buttons and links must respond to Enter and Space
  • Focus order must follow visual reading order (left-to-right, top-to-bottom)
  • Modal/sheet dialogs must trap focus and return it on close
  • Custom widgets (dropdowns, sliders) must implement ARIA keyboard patterns
3

Color Contrast

  • Normal text must meet WCAG AA — minimum 4.5:1 contrast ratio
  • Large text (18px+ bold or 24px+ regular) requires 3:1 minimum
  • UI elements (borders, icons, focus rings) require 3:1 against background
  • Never convey information through color alone — pair with text, icons, or patterns
  • Use the semantic token pairings below for guaranteed-safe contrast
4

ARIA Usage

  • Add aria-label to icon-only buttons and non-text interactive elements
  • Use role="dialog" and aria-modal="true" for sheets and modals
  • Apply aria-expanded to toggles that show/hide content
  • Use aria-live="polite" for dynamic content updates (toasts, loading states)
  • Prefer native HTML elements over ARIA roles — <button> over role="button"
5

Touch Targets

  • Minimum touch target size: 44 × 44px (WCAG 2.5.5 AAA)
  • Sprout inputs use 56px height — exceeding the minimum
  • Icon-only buttons must have sufficient padding to reach 44px
  • Adjacent targets must have at least 8px spacing to prevent mis-taps
  • Small visual elements (checkboxes, radio) should have invisible hit areas that extend to 44px

Safe Contrast Pairings

These token pairings are guaranteed to meet WCAG AA contrast requirements. Use them as your defaults.

Primary on white --text-primary on --bg-primary ~15.4:1
Secondary on white --text-secondary on --bg-primary ~9.6:1
White on brand white on --brand-500 ~4.6:1
White on brand solid white on --bg-brand-solid ~4.5:1
White on red white on --red-500 ~4.6:1
Primary on sky --text-primary on --bg-sky ~13.8:1

Motion & Reduced Motion

Animations must respect user preferences and provide static fallbacks.

Respect prefers-reduced-motion

Wrap all animations and transitions in a @media (prefers-reduced-motion: no-preference) block, or disable them inside @media (prefers-reduced-motion: reduce).

Provide static fallbacks

Every animated state must have a meaningful static equivalent. Loading spinners should fall back to a text label like “Loading…”. Progress bars should display without the fill animation.

Keep durations short

UI transitions should stay under 300ms. Decorative animations should not loop indefinitely — cap at 5 seconds or provide a pause control. Sprout’s standard timings: 0.15s for micro-feedback, 0.2s for toggles, 300ms for sheets.

Accessibility Checklist

Use this interactive checklist to verify your implementation before shipping. Click each item to toggle.

  • Uses semantic HTML elements (button, nav, main, section, table)
  • Heading hierarchy is correct (h1 → h2 → h3, no skipped levels)
  • All interactive elements reachable and operable via keyboard
  • Visible focus indicators on all focusable elements
  • ARIA labels on icon-only buttons and non-text controls
  • All images have descriptive alt text
  • Color contrast meets WCAG AA (4.5:1 text, 3:1 UI elements)
  • Touch targets are at least 44 × 44px
  • Animations respect prefers-reduced-motion
  • No information conveyed by color alone
