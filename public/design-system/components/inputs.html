Input
Inputs allow users to enter and edit text. They support multiple types and states for forms, search, and data entry across Sprout.
  • ✓ For forms that collect names, emails, passwords, ages, or phone numbers.
  • ✓ For search bars that filter or query content.
  • ✗ Don’t use a text input when the user is choosing from a fixed set of options — use a list item instead.

Types

There are five types of text input — Default, Email, Password, Number, and Search.

Default

A general-purpose text field for names, family names, and other free-form text. Uses the default keyboard.

Email

Optimised for email entry. Shows the email keyboard on mobile with @ and . keys. Validates email format and may suggest corrections on error.

Password

Masks the input by default. Always shows a trailing eye icon to toggle visibility. When filled, shows both a checkmark (valid) or error icon and the eye toggle.

Number

For numeric values like age, gem amounts, screen time minutes, or phone numbers. Shows the numeric keypad on mobile.

Search

Includes a leading magnifying glass icon. When focused with a value, shows a clear (✕) button to reset the query. Used for filtering app lists, chats, and block lists.

Anatomy

A text input is composed of up to five parts. Only the container and placeholder/value are required.

  • Label (optional) — Bold text-md above the container. Describes the field.
  • Container (required) — 56px tall, 16px border-radius, 2px border. Changes color per state.
  • Leading icon (search only) — 24px magnifying glass, left-aligned inside the container.
  • Value / Placeholder (required) — Regular text-md. Placeholder in --text-placeholder, value in --text-primary.
  • Trailing icon(s) (contextual) — 24px. Check-circle (valid), alert-circle (error), eye-off / eye-on (password toggle), x-square (search clear).
  • Error message (error state only) — Regular text-sm in --text-error-primary, below the container.

Content guidelines

  • Placeholders should describe the expected value, not repeat the label
  • Keep placeholders short — 2–4 words
  • Use sentence case for both labels and placeholders
  • Error messages should be specific and actionable (e.g. “Invalid email address” not “Error”)
  • For password fields, never reveal the password in error messages

Accessibility

  • Labels: Always associate a <label> with each input via for/id. If using a visual label, it must match. If no visual label, use aria-label.
  • Keyboard: Inputs must be focusable with Tab. The eye toggle and clear button must also be keyboard-accessible.
  • Error states: Use aria-invalid="true" and aria-describedby pointing to the error message element.
  • Autocomplete: Set appropriate autocomplete attributes: email, current-password, new-password, given-name, tel.
  • Contrast: Placeholder text (--text-placeholder) on --bg-primary passes WCAG AA for large text. Value text (--text-primary) passes AAA.
  • Touch targets: The input container at 56px tall exceeds the 44×44px minimum. Trailing icons at 24px should have at least 44px tap area via padding.
