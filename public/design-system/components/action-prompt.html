Action Prompt
Action prompts require active user decision-making. They surface contextual messages with clear calls-to-action that drive toward a resolution — not passive information.
  • ✓ When the user needs to make a decision that moves the experience forward.
  • ✓ For contextual messages tied to a specific event — a completed quest, friend request, or expired timer.
  • ✓ When the message should persist until the user acts or explicitly dismisses it.
  • ✗ Don’t use for passive, read-only information — use a toast or inline text instead.
  • ✗ Don’t use for complex decisions with more than two options — use a sheet.
  • ✗ Don’t use for transient confirmations — use a toast.

Anatomy

An action prompt is a white elevated card with a leading element, text content, an optional close button, and full-width action buttons.

  • Leading element (required) — 46×46px container. Supports TossFace emoji, avatar, or avatar + badge.
  • Label (required) — text-md bold (16px/24px). Frontloads the key information — names, amounts, events.
  • Description (optional) — text-sm regular (14px/20px). Adds context or explains what happens next.
  • Close button (optional) — 12px icon inside 24px tap area. Only shown with single-CTA prompts.
  • Primary button (required) — Full-width Sprout primary button. Action verb label — never generic “OK”.
  • Secondary button (optional) — Dismiss / secondary action. Always paired with primary (never alone).

Action configurations

Buttons always span the full width of the prompt. Choose the configuration based on how the user should respond.

  • Single CTA — One full-width button. Include a close button so the user can dismiss without acting.
  • Dual CTA — Secondary (dismiss) on the left, primary on the right, each taking equal width (16px gap). No close button needed — the secondary button handles dismissal.
  • ✓ CTA labels should start with an action verb: “Claim gems”, “Review”, “Accept”.
  • ✓ Use “Dismiss” as the default secondary label. Use a specific verb when appropriate (“Decline”, “Later”).
  • ✗ Don’t use generic labels like “OK” or “Got it”.
  • ✗ Don’t show a close button alongside dual CTAs — it creates three dismissal paths.

Placement

Position action prompts where they are most relevant to the user’s current activity. They appear inline within content, never pinned to the top of the screen.

Inline — in context
🔔
Bedtime in 10 minutes
Your schedule starts soon.
Screen content
  • ✓ Place on the screen where the related activity is happening (cards page, submissions list, activity feed).
  • ✓ Stack multiple prompts vertically if several require attention.
  • ✗ Don’t pin to the top of the viewport — that pattern is reserved for system-level toasts.

Content guidelines

  • Label: Frontload key information — names, amounts, dates. Keep to one line. Use sentence case, no full stop.
  • Description: Full sentence with a period. Explains context or what happens next. Keep to 1–2 sentences max.
  • Tone: Direct for errors, preventative for warnings, straightforward for everyday tasks, warm for successes.
  • ✓ Be specific: “Ben submitted an exercise proof” not “You have a new submission”.
  • ✓ Match the leading element to the message source, not just the severity.
  • ✗ Don’t use technical jargon — language should work for both parents and kids.

Dismissing a prompt

Action prompts can be dismissed in three ways:

  • Close button — tapping the X icon removes the prompt without acting (single-CTA only).
  • Secondary button — tapping Dismiss / Decline / Later removes the prompt (dual-CTA).
  • Primary action — completing the action resolves and removes the prompt.

Every prompt must have a dismissal path. Never force the user to act.

Accessibility

  • Role: Use role="alert" for destructive prompts. Use role="status" for default prompts.
  • Close button: Must have aria-label="Dismiss" and be keyboard accessible.
  • Action buttons: Use semantic <button> elements. Primary button should receive focus when prompt appears.
  • Focus management: When a prompt is dismissed, return focus to the triggering element or the next logical focus target.
  • Contrast: --text-primary on --bg-primary exceeds 4.5:1. White text on brand-500 and red-500 meets WCAG AA.
  • Motion: If animating entry/exit, respect prefers-reduced-motion.
