Segmented Control
Horizontal segmented control for switching between related views or categories. Uses a pill-in-container pattern with iOS-style sliding transitions.
  • ✓ Use a segmented control when content can be separated into related categories that share the same hierarchy level.
  • ✗ Don’t use a segmented control for sequential steps or navigation between unrelated pages. Use a navigation bar instead.
Interactive · tap to switch

Click any segment to see the iOS-style sliding pill animation.

Variants

Segmented controls come in three display variants depending on the available space and the clarity of the icons used.

Icon + Label

The default variant. Each segment shows a TossFace icon alongside a text label. Use when space allows and labels help users understand the categories.

  • ✓ Use when you have 2–4 segments and enough horizontal space for labels.
  • ✗ Don’t truncate labels. Switch to icon-only if space is tight.
Icon + Label

Icon only

Shows only TossFace icons with no labels. Use when you have many segments (5+) or when the icons are universally understood. Always provide an aria-label for accessibility.

  • ✓ Use for 5+ segments where labels would cause overflow, or for universally understood icons.
  • ✗ Don’t use icon-only if the icons are ambiguous. Labels prevent misinterpretation.
Icon only · 7 segments

Icon + Active Label

Shows icons for all segments but reveals a text label only on the currently selected segment. A good middle ground when you have 3–5 segments and want to confirm the user’s selection without cluttering inactive segments.

  • ✓ Use when 3–5 segments and you want to reinforce which segment is active with a label.
  • ✗ Don’t mix with icon+label. Pick one variant per segmented control.
Icon + Active Label

Layout

Segmented controls support two layout modes that control how segments distribute within the container.

Equal width

Each segment stretches equally to fill the container. Use when you have a fixed number of segments (2–5) and want a balanced, symmetrical appearance.

Equal width · 3 segments
  • ✓ Use equal width when all segments are roughly the same length and the count is low (2–5).
  • ✗ Don’t force equal width on 6+ segments. The touch targets become too narrow.

Content width

Each segment sizes to its content. The container scrolls horizontally with a hidden scrollbar when segments overflow. Use when segment labels vary in length or you have many segments.

Content width · scrollable
  • ✓ Use content width when segment labels vary significantly in length or when you have 5+ segments.
  • ✗ Don’t hide the scroll affordance entirely. Ensure the last visible segment is partially clipped to hint at scrollability.

Animation

Segment switching follows iOS segmented control principles: the active pill slides smoothly to the newly selected segment rather than snapping instantly. This gives spatial continuity and helps users track which segment they selected.

Implementation

For a true iOS-style sliding pill, position a pseudo-element behind the segments and animate its transform: translateX() to the selected segment’s position. This avoids animating width or left (which trigger layout) and runs entirely on the compositor for 60fps.

  • CSS approach: Apply transition: background 0.2s cubic-bezier(0.25, 0.1, 0.25, 1), box-shadow 0.2s cubic-bezier(0.25, 0.1, 0.25, 1) on each segment button. The active background and shadow will crossfade smoothly.
  • Sliding pill approach (advanced): Use an absolutely positioned ::before on the container whose width and transform are set via CSS custom properties (--segment-left, --segment-width) updated by JavaScript on segment change. Animate with transition: transform 0.2s cubic-bezier(0.25, 0.1, 0.25, 1), width 0.2s cubic-bezier(0.25, 0.1, 0.25, 1).
  • Reduced motion: Wrap transitions in @media (prefers-reduced-motion: no-preference) { }. When motion is reduced, the pill should snap instantly.

Overflow & Scrolling

When the segment count exceeds the available width (typically 5+ with labels, or 8+ icon-only), the container scrolls horizontally with a hidden scrollbar.

  • Hidden scrollbar: Use scrollbar-width: none and ::-webkit-scrollbar { display: none } to hide the browser scrollbar while keeping swipe/drag scrolling functional.
  • Clip hint: The last partially visible segment acts as a scroll affordance. Do not add padding that hides this clipping.
  • Scroll-into-view: When a segment is selected programmatically or via keyboard, scroll it into view with scrollIntoView({ behavior: 'smooth', block: 'nearest', inline: 'nearest' }).
  • Touch: Use -webkit-overflow-scrolling: touch for momentum scrolling on iOS.

Accessibility

  • Roles: The container must have role="tablist". Each segment button must have role="tab". Each associated panel must have role="tabpanel".
  • ARIA: Set aria-selected="true" on the active segment, "false" on all others. Link segments to panels with aria-controls="panel-id" and panels back with aria-labelledby="segment-id".
  • Keyboard: Use roving tabindex: the active segment has tabindex="0", all others tabindex="-1". ArrowLeft / ArrowRight move focus between segments. Home / End jump to first/last. Enter or Space activate the focused segment.
  • Icon-only: When labels are hidden, every segment must have an aria-label describing the category (e.g. aria-label="Skin tone").
  • Touch targets: The 44px segment height meets the minimum 44×44px touch target guideline (Apple HIG / WCAG 2.5.8).
  • Motion: All transitions respect prefers-reduced-motion. Wrap animation declarations in @media (prefers-reduced-motion: no-preference).
