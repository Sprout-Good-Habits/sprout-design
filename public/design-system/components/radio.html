Radio
Radio buttons let users choose exactly one option from a set of mutually exclusive choices. Only one radio in a group can be selected at a time.
  • ✓ For mutually exclusive choices where all options should be visible at once.
  • ✓ When the user needs to compare options before choosing (e.g. notification frequency).
  • ✗ Don't use radio buttons when the user can select multiple options — use checkboxes instead.
  • ✗ Don't use radio buttons for on/off toggles — use a switch instead.
  • ✗ Don't use radio buttons for more than 5 options — consider a dropdown or list item instead.

Anatomy

  • Circle (required) — 24px with full radius. Unselected: white with --border-secondary. Selected: --brand-500 fill.
  • Dot (selected only) — 8px white circle centered in the outer circle.
  • Label (required) — Regular text-md in --text-primary. Placed to the right of the circle.
  • Description (optional) — Regular text-sm in --text-secondary. Below the label.

Content guidelines

  • Each option label should be short and distinct from the others
  • Use parallel phrasing across all options in a group
  • Always pre-select a default option unless there is a strong reason not to
  • The group should have a visible question or title above it

Accessibility

  • Role: Use native <input type="radio"> with a shared name attribute, or role="radio" with aria-checked inside a role="radiogroup".
  • Labels: Associate a visible label via <label for> or aria-labelledby.
  • Keyboard: Navigate between options with Arrow Up / Arrow Down. Selection follows focus. Tab moves to the next focusable element outside the group.
  • Groups: Wrap in a fieldset with a legend, or use role="radiogroup" with aria-labelledby.
  • Contrast: The brand-500 selected fill and white dot meet WCAG AA. The unselected border (--border-secondary) on white meets 3:1 for UI elements.
  • Touch targets: The 24px circle is below the 44px minimum — ensure the tap area extends to include the label row (full 370px width).
