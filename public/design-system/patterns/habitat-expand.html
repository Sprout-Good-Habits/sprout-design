When to use

Use Habitat Expand when a single surface is playing two roles and you want the transition between them to feel continuous rather than like a screen swap.

  • ✓ At the end of a habit recording, when the conversational-feedback tile should grow into the full habitat result screen.
  • ✓ When returning from a habitat result back to a live camera screen, collapsing the result into the conversational-feedback tile.
  • ✓ Any place you want Sprout to feel like a persistent character on the surface instead of a character that gets shown and hidden.

When not to use

  • ✗ Don’t use as a generic page transition. This pattern is specifically for tile ↔ habitat.
  • ✗ Don’t use it to move Sprout from one habitat into a different habitat, that is a screen swap, not a morph.
  • ✗ Don’t wrap the tile in a parent that clips or transforms it, the fixed pixel offsets assume the stage fills the 402×874 frame.

Anatomy

The stage is a three-layer stack. From back to front: the habit UI overlay and the morphing habitat card, sitting on top of the gray camera viewfinder inherited from the Video pattern. In the expanded state the card itself is the screen, composing the Background foundation (sky + grass) and the Character foundation (Sprout), so the result scene is just those foundations clipped to the morphing card bounds. Only the classes that do layout or morph work live in /patterns/habitat-expand.css. The home indicator is not part of this pattern, it belongs to the host screen chrome.

  • Stage (required): .habitat-stage is the relative wrapper that controls direction via the .is-tile class. Fills its parent (402×874 in production).
  • Habit UI overlay (required): .habitat-overlay at z-index: 2 holds the camera toolbar, conversational-feedback row, and bottom record bar. Fades with a fast 100ms curve via the .fade-out modifier so the habit chrome drops out before the card geometry moves. Slot markup comes from the Video pattern.
  • Habitat card (required): .habitat-card at z-index: 3 is the morphing element. Default is full-screen sky blue with no radius. Adding .is-tile to the stage shrinks it to the conversational-feedback tile (120×176 at 16, 128, 24px radius, #b9e6fe background). All six transitions (top, left, width, height, border-radius, background-color) share the same 500ms iOS-ish spring curve, cubic-bezier(.32, .72, 0, 1).
  • Sky (required): the card’s expanded fill uses --bg-sky, the same sky token documented in the Background foundation. No new class, the card is the sky surface when expanded.
  • Grass (required): the grass inside the card is the standard grass strip from the Background foundation (grass-edge + grass-fill, --bg-grass), anchored to the card’s bottom edge so it stays glued to the ground as the card grows. It fades to opacity: 0 in the tile state and back to 1 when expanded, over the same 500ms curve.
  • Sprout character (required): the Rive canvas inside the card is a standard Sprout character from the Character foundation, positioned at the canonical grounded-Sprout offset from the Result Screen pattern. The only pattern-local detail is the tile-state translate + scale transform, which shifts Sprout’s face into the tile bounds without changing the layout box. See Canvas inside the card for the math.
  • Title group (optional): .habitat-card-title is positioned inside the card at top: 100px. It is visible in the expanded state with a 500ms delay (so the copy lands right as the 500ms card morph settles), and fast-hides (100ms) the moment the card starts shrinking.

Not part of this pattern: the gray camera viewfinder beneath the card is inherited from the Video pattern, and the home indicator belongs to the host screen chrome. Neither is documented here.

Direction and reversibility

A single class on the stage, .is-tile, drives every layer. Because the CSS is symmetric, flipping the class in either direction yields the opposite transition for free. There is no per-direction CSS or JavaScript branch.

  • Expand: start with .habitat-stage.is-tile, then remove the class after the desired hold. Tile grows to full screen, grass fades in, habit overlay fades out, title fades in 500ms later (right as the card settles).
  • Shrink — start with .habitat-stage (no modifier), then add .is-tile. Full-screen card collapses to the tile, grass and title fade out fast, habit overlay fades back in.
  • The habit overlay is driven separately by adding or removing .fade-out on .habitat-overlay. Add it alongside removing .is-tile (expand); remove it alongside adding .is-tile (shrink).

Trigger example

// Expand after a 1.2s hold
var stage = document.querySelector('.habitat-stage');
var overlay = stage.querySelector('.habitat-overlay');
setTimeout(function () {
  overlay.classList.add('fade-out');
  stage.classList.remove('is-tile');
}, 1200);

// Shrink after a 1.2s hold
setTimeout(function () {
  overlay.classList.remove('fade-out');
  stage.classList.add('is-tile');
}, 1200);

Timing

Every moving piece is on a predictable curve. The card shape, canvas transform, and grass opacity all share one 500ms spring-ish curve, cubic-bezier(.32, .72, 0, 1), which is the closest cubic-bezier approximation to the iOS sheet / matched-geometry spring. The habit chrome and title use shorter durations to stay out of the way.

LayerPropertyDurationEasingDelay
.habitat-card top, left, width, height, border-radius, background-color 500ms cubic-bezier(.32, .72, 0, 1) 0
.habitat-card canvas transform 500ms cubic-bezier(.32, .72, 0, 1) 0
.habitat-card-grass opacity 500ms cubic-bezier(.32, .72, 0, 1) 0
.habitat-overlay opacity 100ms ease-out 0
.habitat-card-title (expand, fade in) opacity, transform 300ms ease-out 500ms
.habitat-card-title (shrink, fade out) opacity, transform 100ms ease-out 0
  • The 500ms title delay on expand is intentional. The card finishes reaching full size at t = 500ms, so the title lands right as the scene settles.
  • The title uses the shorter 100ms fade-out on shrink so the copy gets out of the way before the card geometry moves.
  • The habit overlay uses the shortest fade (100ms) so the habit chrome drops out first on expand, and comes back last on shrink, clearing the way for the card morph to be the main event.
  • Durations and the curve are picked to approximate iOS sheet / matched-geometry spring timing as closely as possible in CSS.

Canvas inside the card

The Rive setup (file, state machine, fit, alignment, inputs) is the standard Sprout character from the Character foundation. The only thing this pattern adds is the tile-state transform, so the same character can sit in the full habitat and in the 120×176 conversational-feedback tile without ever remounting. The trick is: never move the layout box. The canvas stays at a fixed position inside the card, and a single transform switches its visual framing.

  • Canvas layout — position: absolute; left: -144px; top: 380px; width: 690px; height: 690px;. This matches the canonical grounded-Sprout position used by the Result Screen pattern. The box extends beyond the card on the left and bottom, the card clips it.
  • Buffer size — the canvas element uses width="2400" height="2400". A square 2400 buffer renders crisply at 690 CSS px on retina displays.
  • Transform origin — transform-origin: 345px 304px. That is Sprout’s face inside the 690×690 canvas, so scaling happens around the face instead of a corner.
  • Tile transform — .habitat-stage.is-tile .habitat-card canvas { transform: translate(-141px, -596px) scale(.515); }. The scale .515 lines the face up with the tile’s 120×176 box, and the translate slides it into frame. These numbers were tuned to match the conversational-feedback tile exactly.
  • Rive setup — use Fit.Contain and Alignment.BottomCenter so the character grounds on the canvas bottom. This is the same setup as the Result Screen.

Accessibility

  • Reduced motion — under prefers-reduced-motion: reduce, all card, canvas, grass, title, and overlay transitions are disabled via transition: none !important. The state toggle still applies instantly, so the pattern still works as a cross-fade swap.
  • Focus — the morph itself is presentational. Put focusable controls (close button, continue button) inside the overlay or title group and let them inherit normal focus order.
  • Announcements — if the expanded end state is effectively a Result Screen, the title group should be announced via the usual Result Screen mechanism (e.g. role="status" with aria-live="polite"), not by the morph wrapper.

Relationships

  • Result Screen — the expanded end state is visually identical to a Result Screen. The canvas position, grass strip, and title zone are the same numbers.
  • Video and Camera — the tile state lives on top of a camera view, so the habit overlay uses the same slot layout documented there.
  • Conversational Feedback — the tile state is the conversational-feedback tile. The 120×176 dimensions, 24px radius, and #b9e6fe background all come from that component.
  • Character — see the full Rive input reference for which triggers to fire at each end state.
