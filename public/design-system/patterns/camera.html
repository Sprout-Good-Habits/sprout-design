Camera
Full-screen photo capture with Sprout guiding the user through framing and shooting. Pairs a floating conversational feedback card with a dedicated Button Recording control and two Button Utility buttons for camera flip and flash.
  • ✓ Use the Camera pattern when you need the user to capture a still photo and you want Sprout present as a coach.
  • ✗ Don’t use for video or audio capture. Use the Video pattern instead.

When to use

Reach for the Camera pattern whenever a kid needs to take a still photo to finish a task, prove a habit, or share their work.

  • ✓ Capturing a drawing, a homework sheet, or a proud moment.
  • ✓ Quick habit check-ins (for example, “Snap your finished chore”).
  • ✓ Profile photo or avatar selection flows.

When not to use

  • ✗ Continuous recording or timed performances. Use the Video pattern.
  • ✗ Pure document scanning that doesn’t need a character guide. Use a plain system camera sheet.
  • ✗ Photo selection from the gallery. Use a dedicated picker component.

Anatomy

The Camera pattern is a full-bleed viewfinder with UI chrome floating on top as absolutely positioned overlays. Heights assume a 402×874 iPhone mockup. Layer order from bottom to top:

0. Viewfinder (full bleed, edge-to-edge): The live camera feed fills the entire device screen. In docs it is shown as a neutral #d9d9d9 placeholder. Every other element floats on top.
1. Status bar (62px): System chrome passed through from the device. Pinned to the top of the screen.
2. Toolbar (top 62px): A Toolbar component composing a Button Utility close. Three variants are valid here, all built on the Button Utility slot pattern. Pick whichever fits the host screen:
  • toolbar-quest-transparent (Quest Overlay): transparent background with a btn-util close and a Progress Bar. Best default for a full-bleed camera view.
  • toolbar-default: leading slot holds a btn-util close, optional absolute centered title, optional trailing slot for a primary action. Use when the capture is embedded inside a labelled flow.
  • toolbar-title-2-line: leading and trailing btn-util slots with a centered two-line title. Use when the parent flow needs a title plus subtitle above the viewfinder.
The demo above uses toolbar-quest-transparent. Never use toolbar-quest (the non-overlay variant): its sky-blue background clashes with the viewfinder.
3. Conversational feedback (top 126px, inset 16px): A Conversational Feedback component in the Conversational type, Incoming direction. Uses an avatar-portrait on the left and subtitles-overlay on the right so the bubble reads clearly over the viewfinder. The character is either Sprout (.rive-sprout) or the Village Character (.rive-user) depending on who is coaching.
4. Bottom bar (72px, pinned 34px from bottom): A space-between row of Flip camera (48×48 Button Utility), the Button Recording Secondary Start (72×72 shutter, white with a neutral indicator dot), and Flash toggle (48×48 Button Utility). Swap the shutter to Primary Start only if the camera is embedded in a flow that needs the stronger blue fill.
5. Home indicator (134×5, pinned 8px from bottom): Standard iOS home bar.

Composition

Every element on screen is an existing component. The pattern owns only the layout and the state transitions.

Toolbar
Quest Overlay variant with a Button Utility close and a nested Progress Bar. See Anatomy for the full list of valid toolbar variants.
Button Utility
48×48 icon button used three times: Close in the Toolbar, Flip camera, and Flash toggle. Icons are inline SVG (camera-rotate and zap).
Progress Bar
Nested inside the Toolbar. Represents the kid’s progress through a multi-step quest, never elapsed capture time.
Conversational Feedback
Conversational type, Incoming direction. Pairs an avatar-portrait with subtitles-overlay so the bubble reads against the viewfinder. Drop in Sprout or the Village Character.
Button Recording
Secondary Start (white with a neutral indicator dot) is the default shutter so it stays out of the way of the viewfinder. Swap to Primary Start only if the capture needs the stronger brand blue fill.

Character’s role

The guiding character (either Sprout or the Village Character) is the camera coach. Copy should be concrete and action-oriented, matching the tone of the host flow. Keep each line to one or two short sentences so it fits the feedback card without truncation.

  • Ready state: “Show me your drawing. Hold the camera steady and tap the shutter.”
  • Framing help: “A bit closer so I can see every detail.”
  • Review state: “Want to keep this one or try another?”
  • ✓ Match the host flow’s tone: neutral instruction for utility captures, warmer for celebration or practice moments.
  • ✓ Tell the kid what to do next, not what just happened.
  • ✗ Don’t describe the button with words like “tap the camera icon on the right”. Trust the shutter.
  • ✗ Don’t force encouragement. The pattern is neutral by default and only reads as positive when the host flow calls for it.

States

The Camera pattern has two top-level states. Keep Sprout’s copy in step with the state change so the kid always knows what comes next.

  • Ready: The viewfinder is live, the shutter is enabled, and Sprout prompts the kid to capture.
  • Review: The captured still fills the viewfinder. Sprout asks the kid to keep or retake. Swap the bottom bar for a pair of Buttons (Primary “Keep” + Secondary “Retake”).

Accessibility

  • Live region: Wrap the Conversational Feedback in role="status" aria-live="polite" so Sprout’s coaching copy is announced when it changes.
  • Shutter label: Give the Button Recording aria-label="Take photo". In the Review state, keep the bottom bar but update labels to “Keep photo” and “Retake”.
  • Touch targets: The 72×72 shutter and 48×48 utilities both exceed the 44×44 minimum. Keep the bottom bar 16px away from the screen edges.
  • Reduced motion: Sprout’s Rive animation respects prefers-reduced-motion via the shared init script. Do not layer additional transitions on the viewfinder.
