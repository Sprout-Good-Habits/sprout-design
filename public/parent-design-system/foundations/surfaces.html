Surfaces

How we think about the visible boxes in parent chat. Six surface families, distinguished by where they sit in the z-stack and how they elevate off the canvas.

The six families

Every surface in parent-chat-3 belongs to one of these. Pick the family that matches the role first, then borrow its specs. New surfaces should fit an existing family before inventing a new one.

Floating vs inline: when to pick which

Floating and inline cards both use the white primary background, but they signal different things. The choice is about whether the surface is part of the conversation or sits above it.

Floating
Lifted off canvas
Soft shadow, no border. Use when the surface needs to feel separate from the thread (composer, toolbar, banners). It belongs to the chrome, not the conversation.
Inline
Sits in flow
1px border, no shadow. Use when the surface is part of the conversation (confirm-card, artifact-card). It scrolls with the messages.

Sheets are a pattern, not a primitive

The three bottom sheets (reasoning, artifact, context-menu) all share the same anatomy. Treat this as one pattern when adding a new sheet, not three independent components.

  • Anchored bottom, transforms in from translateY(100%)
  • Top corners only: 24px radius (the bottom is flush with the screen edge)
  • Scrim behind: --bg-overlay at z-index just below the sheet
  • Horizontal padding: 16px (--spacing-xl) on header, body, and footer
  • Bottom padding: at least 32px (--spacing-4xl), 40px (--spacing-5xl) when the last child is interactive (safe-area for the home indicator)
  • Drag handle (optional): 36 by 4px, --gray-300, centered above the content

What about radius and shadow?

Radius values pull from the canonical scale at kid-design-system / radius. Most surfaces here use --radius-2xl (16px), --radius-3xl (20px), or --radius-4xl (24px). Pills go straight to --radius-full (9999px) or large explicit values like 296px.

Shadows are not yet tokenized. Floating cards reach for low-alpha inline values like 0 0 8px rgba(0,0,0,.04) on the toolbar pill or 0 1px 8px rgba(0,0,0,.06) on banners. Sheets do not use a box-shadow at all (they sit on the scrim, which provides the visual separation). See drift below.

Drift to flag

What's in the code today that does not yet fit the system. Documenting these so we can tighten them later, not blocking shipping on them.

Open questions
  • Floating-card shadows are inline. chat-toolbar, chat-banner, and chat-composer each declare their own low-alpha box-shadow. Three values, no shared token. Candidate for a small shadow scale (e.g. --shadow-floating).
  • Toolbar pill radius is 296px. A magic number that doesn't match the radius scale. Either accept it as an intentionally-very-rounded one-off or snap to --radius-full.
  • Sheet padding-bottom fallback is wrong. Sheets declare padding-bottom: var(--spacing-4xl, 40px), but --spacing-4xl resolves to 32px (the 40 fallback was probably copy-pasted from --spacing-5xl). Resolved value is correct; fallback is misleading.
  • "Card" means three things (floating, inline, sheet). The CSS classes do not encode that distinction. New code should pick the family by name, not by the word "card".
  • Plus-pill background uses --bg-tertiary (gray) while suggestion/option chips use --brand-100 (light blue). Intentional split (plus-pill is utility, chips are content), worth documenting somewhere visible.
