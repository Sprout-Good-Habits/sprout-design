Cards

28 different card components live in the catalog. Behind them are three reusable lenses (structure, surface, intent) that combine into the cards we ship. Pick from each lens, get the right card.

The three lenses

Every card answers three questions: what shape, what skin, what mood. Picking one value from each lens gives you a card. Existing cards are not arbitrary; each one is a specific (structure, surface, intent) triple.

Recipes: the most common combos

Out of 5 × 5 × 7 possible triples, only ~10 actually appear in the catalog. They tend to cluster. These are the "recipes" worth standardizing.

Recipe 1
The status row
Row + Default + Neutral + status pill
Surface a thing with one piece of state. Saved skills, contacts, documents, sent confirmations.
Recipe 2
The data list
List + Default + Neutral
Structured key-value rows. Schedules, exports, tracking logs.
Recipe 3
The celebration
Stat + Tonal + Success or Knowledge
Quiet wins. Milestones, completions, sent confirmations.
Recipe 4
The result banner
Bannered + Bordered semantic + Success or Destructive
Search hit, share preview, error explanation. Header band sets the topic, body delivers details.
Recipe 5
The decision card
Action + Bordered strong + Warning or Destructive
Anything that needs an explicit yes/no. Confirm cards for destructive or permission-granting actions.
Recipe 6
The outbound message
Stat + Bordered semantic + Action (brand)
Drafts and replies. Anything Sprout is sending or receiving on behalf of the parent.

Picking a card: the decision flow

  1. What's the card for? Inform (default), prompt action (action), celebrate (tonal), warn (semantic), confirm something irreversible (strong).
  2. How much info? One row → Row. Title + blurb → Stat. Many comparable rows → List. Big visual moment → Bannered. Decision required → Action.
  3. What intent does it carry? Pick one of the seven colors (neutral / action / knowledge / trust / warning / success / destructive). The intent decides the surface color.
  4. How urgent is it? Default for neutral. Bordered semantic when the intent matters but the action isn't critical. Bordered strong only when the parent must explicitly consent or the action can't be undone. Tonal when celebrating or softening a warning.

If you can't make those four choices easily, the card probably doesn't need to exist as a separate component. Reach for a simpler primitive (a message bubble + a paragraph, an action button row, a notif-banner).

Drift to flag

Open questions
  • Border weights are inconsistent. Most cards use 1px. coppa-card and confirm-card use 2px. The 2px choice signals "critical" but isn't tokenized. Candidate: --border-strong-width.
  • Tonal background colors don't agree. grad-attach uses sprout-25, completion-card and sent-card use green-50, retry-block.succeeded uses green-50. Knowledge tonal vs success tonal is the right split, but it's not labeled.
  • Dashed border is a one-off. Only skill-attach uses it (knowledge entity). Either promote "Dashed" to a real surface family or migrate skill-attach to bordered semantic + the knowledge-tag pill.
  • Bannered cards re-implement the same pattern. result-attach, share-card, export-card each declare their own header band CSS. Candidate: shared .card-banner primitive.
  • Body font sizes drift. 10, 11, 12, 13px all appear inside cards. The detail tier (13px) is also called out in type.
  • Action button styling forks. confirm-card's .confirm-card-btn and the standalone .msg-btn do roughly the same thing with different class names. Candidate: unify on .btn-stack.
