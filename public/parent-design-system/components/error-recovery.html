Error Recovery

Errors are something Sprout says, not a system alert. Every error is a conversational message that explains what happened and, when the parent needs to act, pulls in the right card or buttons.

Design principle

Error recovery is not a component. It is a conversational pattern: Sprout tells the parent what happened in natural language, as though they are having a conversation. The parent should never feel like they hit a dead-end system error. They should feel like Sprout noticed something, explained it, and is helping them through it.

  • Contextual, not generic: Sprout names the specific thing that failed ("I couldn't reach Google Calendar") rather than a vague label ("Something went wrong").
  • Natural language: Written in first person, the way Sprout talks. Calm, specific, and helpful.
  • Action via cards and buttons: When the parent needs to do something, Sprout attaches the relevant card (connector card, confirm card) or button stack below the message. No action required? Just the message bubble, nothing extra.

When to use

  • Transient failures: A service is temporarily down, a request timed out. Sprout explains and may offer to retry or let the parent send manually. No card needed, just the message.
  • Action required: The parent needs to re-authenticate, update a password, grant a permission, or reconnect a service. Sprout explains why and attaches the relevant card or buttons.
  • Permission denied: Missing scope or consent. Sprout names the missing permission and provides a button or card to fix it.
  • Data mismatch: What Sprout received does not match what it expected. Sprout explains the discrepancy and suggests next steps.

Anatomy

Error recovery uses the standard Sprout Message anatomy. There is no special error wrapper or error-specific component.

  • Sprout avatar + bubble: The error explanation lives in a regular .msg-bubble-in. Sprout's tone is calm and specific.
  • Card (optional): When the parent needs to reconnect a service, a Card is attached as a sibling inside .msg-col.
  • Button stack (optional): When the parent needs to choose an action (e.g. "Update password" / "Log out"), a .btn-stack is attached below the bubble.
  • Message actions: Standard copy and feedback buttons remain.

Examples

Temporarily unavailable (no action required)

The service is down but Sprout will handle it. The parent just needs to know.

Sprout
ClassDojo is a little slow right now, so I wasn't able to pull Emma's latest report. I'll try again in a few minutes and let you know when it's ready.

New password needed (action required with buttons)

The parent's saved credentials expired. Sprout explains and provides action buttons.

Sprout
It looks like your school portal password was changed recently, so I can't log in anymore. You'll need to update it so I can keep pulling Noah's assignments.

Permission denied (action required with connector card)

Sprout needs a scope the parent hasn't granted. The connector card lets them fix it in one tap.

Sprout
I don't have permission to read your Google Calendar yet. If you connect it, I'll be able to check for scheduling conflicts before suggesting activity times.
Google Calendar
Read your schedule

Reconnection needed (action required with connector card)

A previously connected service lost its connection. Sprout explains why and surfaces the reconnect card.

Sprout
Your school portal connection expired. This happens every 90 days for security. Once you reconnect, I'll catch up on any missed assignments.
School Portal
Session expired

Content guidelines

Do "ClassDojo is a little slow right now. I'll try again in a few minutes."

"Your school portal password was changed, so I can't log in anymore."

"I don't have permission to read your Google Calendar yet."
Don't "Error: Service unavailable (503)"

"Authentication failed. Please re-enter credentials."

"Permission denied. Missing scope: calendar.readonly"
  • Name the service: Say "Google Calendar" not "the service."
  • Explain the impact: Tell the parent what this means for them ("I can't pull Noah's assignments").
  • Offer the fix: If the parent can do something, surface the right card or button. If not, tell them what Sprout will do ("I'll try again in a few minutes").
  • No codes or jargon: Never show HTTP status codes, technical error names, or developer language.

Decision tree

  • Can Sprout fix it automatically? Message only, no buttons. ("I'll retry in a few minutes.")
  • Does the parent need to reconnect a service? Message + connector card with Reconnect/Connect button.
  • Does the parent need to take a non-connector action? Message + button stack (e.g. "Update password" / "Remind me later").
  • Is the error after exhausted retries? Follows a Retry Block, then this message surfaces next steps.
