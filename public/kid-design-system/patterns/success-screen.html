Result Screen
Full-screen pattern for quest and task outcomes. Communicates completion status to the kid with an encouraging tone, a large Sprout character, and a single Continue action. Used for approved completions, partial completions, and pending parent review.

When to use

Use a result screen to communicate the outcome of a completed quest or task to the kid.

  • ✓ After a quest reaches readiness = 1.0 (full completion).
  • ✓ After a quest is auto-approved above the parent's threshold (partial completion).
  • ✓ After a quest is sent to parent for review (below threshold or expired).
  • ✓ Any kid-facing end state that needs to feel warm and encouraging.

When not to use

  • ✗ Don't use for error states — use the Error Screen pattern.
  • ✗ Don't use for parent-facing review screens — those use the Parent Review layout.
  • ✗ Don't use for in-progress states (readiness bar, camera view).

Anatomy

The Result Screen is a composition, not a bag of new components. It reuses the Background foundation (sky + grass), the Character foundation (Rive canvas), and the Primary Button component. The only pattern-specific pieces are .result-screen (the wrapper) and .result-content (the title + description + optional stat-card zone). Both live in /patterns/success-screen.css.

  • Wrapper (pattern) — .result-screen is the only structural class the pattern owns. It’s a full-bleed var(--bg-sky) container with position: relative and overflow: hidden. Every other piece is a known foundation or component placed inside it as a direct child.

  • Character (foundation) — a regular Rive Sprout canvas, same element documented in Character. The pattern positions any <canvas> placed as a direct child of .result-screen at 690×690, bottom: -196px, horizontally centered (the same “Chat (Tier 2)” placement documented in the Character foundation). Use Fit.Contain + Alignment.BottomCenter so the feet ground on the grass.
  • Grass (foundation) — the same .grass-container / .grass-edge / .grass-fill markup documented in Background. The pattern pins it to bottom: 0 at z-index: 1 with the Main variant’s 89px grass-fill height.
  • Action (component) — a regular Primary Button, .btn-sprout.btn-primary.btn-lg. No wrapper. The pattern positions any .btn-sprout placed as a direct child of .result-screen at bottom: 45px, full-width (left: 16px; right: 16px).

With stats

Stat cards are an optional layer. When a quest produces both a measurable performance number (accuracy percentage or practice count) and a session duration, show them as a pair of Stat Cards directly under the description. The two cards reinforce and visually anchor numbers the description already mentions, they do not replace the description.

  • ✓ Use stats when the quest has both a Target value (accuracy, count) and a Duration (session time).
  • ✓ Place the row inside .result-content, directly below .result-desc, using .stat-card-row. Cards flex to equal width.
  • ✗ Don’t stack cards vertically, they must sit side by side.
  • ✗ Don’t show stats on the “Sent to Parents” variant, the neutral review tone does not pair with celebratory numbers.
  • ✗ Don’t override the colour pairing. Target is always yellow, Duration is always green.

Counter animation

When stat cards appear on a result screen, their numeric values count up from 0 to the final value. This makes the moment feel like a reveal and gives the Rive celebration a beat to land before the numbers settle. The animation is split into two phases:

PhaseDurationEasingDelayNotes
Card entry (scale + fade) 350ms Spring cubic-bezier(0.5, 1.8, 0.3, 0.8) 200ms for Target, 300ms for Duration Matches the project spring curve. Implemented in CSS via @keyframes stat-card-pop.
Counter tween (0 to final) 600ms easeOutCubic (1 - (1 - t)^3) 550ms for Target, 650ms for Duration Starts right after the card entry finishes. Spring is intentionally not used on numbers, overshoot makes a value briefly flash past its target.

Format rules

Each .stat-card-number declares its final value and format with three data attributes. The shared helper at /components/stat-card.js reads them:

  • data-count-to="N", the final numeric value. For time, this is total seconds.
  • data-count-format="percent", tween an integer and append %. Use for Target cards. Example: data-count-to="80" renders 80%.
  • data-count-format="time", tween total seconds and format as M:SS. Use for Duration cards. Example: data-count-to="600" renders 10:00.

Usage

Mark up the cards with the data attributes, then call SproutStatCard.animateRow(rowEl) when the result screen mounts. The helper handles delay, stagger, formatting, and prefers-reduced-motion.

<div class="stat-card-row">
  <div class="stat-card stat-card--target">
    <div class="stat-card-label">GREAT</div>
    <div class="stat-card-value">
      <span class="stat-card-icon">&#127919;</span>
      <span class="stat-card-number"
            data-count-to="80"
            data-count-format="percent">80%</span>

  </div>

Reduced motion

If the user has prefers-reduced-motion: reduce set, both phases are skipped. The CSS entry animation is disabled by a media query, and SproutStatCard.animateCount writes the final formatted value immediately and returns before scheduling the tween. The final visual state is identical, only the motion is removed.

Sprout behavior

Sprout's expression and speech are driven by the Rive state machine and vary by outcome.

VariantRive triggerMouth animationTone
Full Completion celebration Speaks the description text (1700ms delay after celebration) Celebratory, proud
Partial (Above Threshold) celebration Speaks the description text (1700ms delay after celebration) Encouraging, positive
Sent to Parents smile Speaks the description text (200ms delay after smile) Reassuring, warm
  • Mouth animation duration is calculated from the description word count: max(1500ms, wordCount × 250ms).
  • A hidden .subtitle-card p element holds the text that drives the mouth animation timing.
  • The celebration trigger must complete before the mouth starts — the 1700ms delay ensures this.
  • Never show a sad or negative Sprout expression on result screens. Even “Sent to Parents” uses smile for an encouraging tone.

Copy guidelines

The description text is what Sprout “says” to the kid. Write it in first person from Sprout’s voice.

  • Full Completion — Celebrate the specific achievement. Mention what they did and how well. “I'm so proud of you!”
  • Partial — Acknowledge effort, note what was done, encourage next time. “So close!”
  • Sent to Parents — Explain what happened without blame. Use “we” language. “We'll hear back from them soon!”
  • Keep descriptions under 25 words for comfortable mouth animation timing.
  • Never frame the outcome as failure. Every result screen is a positive moment.

Rive character

The Sprout character uses sprot2.97.riv with state machine State Machine 1.

  • File: sprot2.97.riv — hosted at https://sprout-hq.vercel.app/assets/sprot2.97.riv
  • Canvas: 1380×1380 internal resolution, displayed at 690×690 CSS pixels
  • Fit: Fit.Contain, Alignment.BottomCenter
  • Positioning: bottom: -196px, centered horizontally — character is grounded and clipped by the grass edge
  • Default customization: hairID = 1 (short hair)

Available triggers for result screens

  • celebration — Full-body bouncy animation. Use for positive outcomes (completion, approval).
  • smile — Warm encouraging expression. Use for neutral-positive outcomes (sent to parents).
  • characterwave — Wave + smile. Could be used for “see you next time” type outcomes.
