Hierarchy

Hierarchy sets a visual ranking amongst the buttons displayed on the screen to help more important buttons take precedence over others. For third-party sign-in flows, see Social Buttons.

Primary Positive

The most important action to move forward in a flow, acknowledge and dismiss, or finish a task. You should always have one primary button per page where relevant.

  • ✓ Use one primary button per page to indicate the most important action.
  • ✗ Don’t use multiple primary buttons on a page.

Sizes

There are four button sizes — large, medium, small, and extra small — each used for different purposes.

Large

The large button is used to move users forward in a flow. It’s used at full width on mobile. Large supports Toss Face icon left.

Large · 48px · 18px
  • ✓ Use large button at the bottom of the page or content to aid the user to the next step.
  • ✗ Don’t use large buttons inside cards or list items.

Medium

The medium button is used for inline content. Medium supports Toss Face icon left. Use inline with other buttons at the same size.

The small button is used for card actions, compact forms, and tighter layouts. It uses a 12px border radius and 14px font to stay proportional. Small supports Toss Face icon left.

The extra small button is a compact, content-width button for inline actions inside list items and dense layouts. It uses an 8px border radius and content-fit width, signaling a different context from the full-width sizes.

Icons

Use icon left to support the message with a Toss Face icon. Use icons that best match the written content.

  • ✓ Use an icon that reinforces the button’s label and helps convey its meaning at a glance.
  • ✗ Don’t use icons for the sake of it — only add one when it genuinely aids comprehension.

Interaction

Unless the button is disabled, the user can tap or click on it to perform the action assigned to it.

Placement

The button should always be contextual, which is why it should be used inline with the content. This gives a stronger cohesion between the context and the action that it performs.

On mobile, if an action must be taken by the user — for example, to finalise something or proceed in the current process — it is best to keep the button visible all the time. You can do this by pinning it to the bottom of the screen, outside of the scrolling area. Please pin only one primary action per screen.

Content

Button copy should:

  • Start with a verb, like “Play” or “Send”
  • Be just a few words (ideally 1 or 2)
  • Describe the action — if someone only reads the button, they should know what will happen next
  • Connect to the content around it (for example, it might use the same word as the title)
  • Avoid using first person like “me”, “my”, and “I”
  • Be in sentence case (only capitalise the first letter of the first word)
  • Have no full stop

Accessibility

Sprout is a kids app, but parents and educators use it too. All buttons must meet WCAG AA standards.

  • Keyboard: All buttons are focusable and activated with Enter or Space. Use native <button> elements — never <div> with click handlers.
  • Focus ring: Buttons show a visible 2px focus ring on keyboard navigation. Do not remove :focus-visible outlines.
  • ARIA: Use aria-disabled="true" instead of the disabled attribute when the button should still be discoverable by screen readers. Add aria-busy="true" during loading states.
  • Contrast: Primary buttons (white on --brand-500) pass AA for large text. Secondary buttons (--brand-500 on white) pass AA at all sizes. Disabled buttons are exempt from contrast requirements.
  • Touch targets: Large and medium sizes (44px, 48px) meet the 44×44px minimum touch target guideline. Small (36px) is close and should be used with sufficient surrounding spacing. Extra small (28px) relies on surrounding padding in its list-item context to meet the 44×44px tap area — ensure at least 8px padding around extra small buttons.
  • Motion: The press animation (translateY) respects prefers-reduced-motion — wrap transitions in a media query.
